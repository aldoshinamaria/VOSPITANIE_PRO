# Normative Center Specification

## 1. Назначение Нормативного центра

Нормативный центр Воспитание.PRO — раздел системы для хранения, учета, проверки и обновления нормативных документов, которые влияют на воспитательную работу школы.

Его задача — помочь заместителю директора по воспитательной работе понимать:

- какие документы используются при формировании рабочей программы воспитания;
- какие документы влияют на КПВР;
- какие нормативные источники актуальны;
- какие документы требуют проверки или обновления;
- какие расхождения есть между школьными документами и требованиями более высокого уровня.

Нормативный центр не должен автоматически изменять рабочую программу, КПВР, мероприятия или другие operational data. Он предоставляет источники, статусы, предупреждения и рекомендации. Любые изменения школьных документов выполняются только после проверки и подтверждения пользователя.

## 2. Какие документы хранятся

### Федеральные документы

Федеральные документы имеют наивысший приоритет среди образовательных источников.

Примеры:

- федеральное законодательство;
- ФГОС;
- ФОП;
- федеральная рабочая программа воспитания;
- федеральный календарный план воспитательной работы;
- рекомендации Минпросвещения;
- федеральные стратегии и концепции.

### Региональные документы

Региональные документы уточняют федеральные требования с учетом субъекта РФ.

Примеры:

- региональные приказы;
- письма и методические рекомендации;
- региональные планы воспитательной работы;
- региональные программы развития воспитания;
- документы региональных операторов.

### Муниципальные документы

Муниципальные документы отражают требования и планы муниципального уровня.

Примеры:

- муниципальные приказы;
- планы мероприятий;
- рекомендации управления образования;
- положения о конкурсах и акциях;
- муниципальные дорожные карты.

### Локальные акты школы

Локальные акты школы фиксируют, как требования реализуются в конкретной образовательной организации.

Примеры:

- рабочая программа воспитания школы;
- КПВР;
- положения о школьных объединениях;
- приказы директора;
- планы работы объединений;
- планы профилактики;
- планы внеурочной деятельности;
- отчеты о воспитательной работе.

## 3. Категории документов

Нормативный центр должен поддерживать следующие категории документов:

- ФГОС;
- ФОП;
- Федеральная рабочая программа воспитания;
- Федеральный календарный план;
- стратегия;
- приказы;
- положения;
- планы;
- отчеты;
- иные документы.

Каждый документ должен иметь:

- id;
- название;
- уровень документа;
- категорию;
- дату;
- версию;
- источник;
- статус актуальности;
- дату загрузки;
- связь с обработанным документом, если он прошел document-processing;
- признаки классификации, если они были определены rule-based classifier.

## 4. Правила актуальности документов

### current

Статус `current` означает, что документ считается актуальным и может использоваться как источник требований.

Условия:

- документ имеет дату и версию;
- источник понятен;
- документ не заменен более новой версией;
- документ не помечен как устаревший;
- пользователь подтвердил его актуальность или система уверенно определила статус по правилам.

### needs_review

Статус `needs_review` означает, что документ требует проверки пользователем.

Причины:

- не определена дата;
- не определена версия;
- источник указан неполно;
- документ похож на нормативный, но классификация низкоуверенная;
- найден возможный более новый документ;
- документ противоречит другому источнику;
- документ был извлечен из файла, качество которого требует проверки.

### outdated

Статус `outdated` означает, что документ не должен использоваться как актуальный источник требований без ручной проверки.

Причины:

- существует более новая версия;
- истек срок действия;
- документ отменен или заменен;
- пользователь вручную отметил документ как устаревший;
- документ противоречит источнику более высокого приоритета.

## 5. Правила версионирования документов

Каждый нормативный документ должен иметь версию или способ ее определения.

Версионирование должно фиксировать:

- documentId;
- versionId;
- дату документа;
- номер документа, если есть;
- дату загрузки;
- источник;
- статус актуальности;
- предыдущую версию;
- связь с заменяющим документом;
- список затронутых разделов системы.

Правила:

- новая версия документа не должна автоматически удалять старую;
- старая версия должна получать статус `outdated` или `needs_review`;
- пользователь должен видеть, какой документ заменил предыдущий;
- проверки соответствия должны использовать только актуальные или подтвержденные документы;
- если версия не определена, документ должен быть помечен как `needs_review`.

## 6. Правила связи документов с сущностями системы

### Связь с рабочей программой воспитания

Нормативные документы используются для:

- проверки обязательных разделов;
- проверки федеральных направлений воспитания;
- проверки целевых ориентиров;
- проверки актуальности формулировок;
- формирования рекомендаций по обновлению программы.

Рабочая программа не должна автоматически переписываться при загрузке нового нормативного документа. Система должна показать расхождения и предложить пользователю обновить нужные разделы.

### Связь с КПВР

Нормативные документы используются для:

- проверки наличия федеральных дат;
- сверки КПВР с федеральным календарным планом;
- выявления отсутствующих обязательных или рекомендованных мероприятий;
- проверки соответствия уровней образования;
- формирования предупреждений о неполном КПВР.

Федеральный календарный план не должен автоматически создавать мероприятия в `SchoolEvent` без preview, validation, duplicate detection и user confirmation.

### Связь с мероприятиями

Документы могут быть источником:

- preview-мероприятий;
- рекомендаций по добавлению мероприятий;
- признаков для проверки дублей;
- source metadata для импортированных событий.

Любое мероприятие, извлеченное из документа, должно проходить безопасный импорт:

1. preview;
2. validation;
3. duplicate detection;
4. user confirmation.

### Связь с проверкой соответствия

Нормативный центр предоставляет Compliance Engine:

- список актуальных документов;
- приоритет источников;
- категории требований;
- статусы актуальности;
- предупреждения о недостающих документах;
- признаки устаревших документов.

Compliance Engine должен учитывать приоритет документов:

1. федеральное законодательство;
2. ФГОС;
3. ФОП;
4. федеральная рабочая программа воспитания;
5. федеральный календарный план;
6. рекомендации Минпросвещения;
7. региональные документы;
8. муниципальные документы;
9. локальные акты школы.

Документы низшего уровня не должны отменять требования документов более высокого уровня.

## 7. Правила отображения пользователю

Пользователь должен видеть не просто список файлов, а управляемую нормативную базу.

Для каждого документа отображать:

- название;
- уровень;
- категорию;
- дату;
- версию;
- источник;
- статус актуальности;
- дату загрузки;
- качество обработки;
- классификацию документа;
- связи с рабочей программой, КПВР или проверкой соответствия;
- предупреждения.

Основные пользовательские блоки:

- актуальные документы;
- требуют проверки;
- устаревшие документы;
- федеральная база;
- региональная база;
- муниципальная база;
- локальные акты школы.

Пользователь должен понимать:

- какие документы уже есть;
- каких документов не хватает;
- какие документы устарели;
- какие документы влияют на рабочую программу;
- какие документы влияют на КПВР;
- что нужно обновить в первую очередь.

## 8. Правила обновления нормативной базы

Обновление нормативной базы должно быть безопасным и обратимым.

Правила:

1. Новый документ сначала загружается и проходит document-processing.
2. Система определяет тип, уровень, категорию и уверенность классификации.
3. Если уверенность низкая, документ получает статус `needs_review`.
4. Если найден похожий документ, система показывает возможное обновление версии.
5. Старая версия не удаляется автоматически.
6. Пользователь подтверждает актуальность нового документа.
7. После подтверждения документ может использоваться в проверке соответствия.
8. Рабочая программа, КПВР и мероприятия не изменяются автоматически.
9. Система показывает, какие разделы документов школы могут потребовать обновления.

Нормативный центр должен поддерживать будущую AI-интеграцию через заменяемые контракты анализа, но на текущем этапе использовать rule-based подход и подтверждение пользователя.

## Ограничения MVP

На этапе MVP Нормативный центр должен обеспечивать:

- понятное хранение документов;
- классификацию документов;
- статусы актуальности;
- связь с проверкой соответствия;
- предупреждения о недостающих или устаревших источниках;
- безопасную подготовку к обновлению рабочей программы и КПВР.

Не нужно преждевременно добавлять:

- автоматическую замену школьных документов;
- скрытое обновление мероприятий;
- автоматическое утверждение нормативных требований;
- внешний AI-анализ без контрактов и проверки пользователя.

Главное правило: Нормативный центр является источником требований и контроля актуальности, но не должен без подтверждения пользователя изменять operational data школы.
