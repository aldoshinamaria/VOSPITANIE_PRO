# KPVR Builder Specification

## 1. Назначение KPVR Builder

KPVR Builder — механизм автоматической сборки календарного плана воспитательной работы школы на основе единой модели данных Воспитание.PRO.

Его задача — формировать КПВР как проекцию существующих мероприятий, а не как отдельный набор скопированных записей. Основным источником остается `SchoolEvent`. КПВР должен автоматически отражать актуальные мероприятия школы, распределять их по уровням образования, модулям воспитания и месяцам, а также сохранять трассировку происхождения данных.

KPVR Builder должен поддерживать:

- отдельные планы для НОО, ООО и СОО;
- группировку по модулям воспитания;
- календарное распределение по месяцам;
- учет федеральных дат;
- предотвращение дублей;
- проверку полноты;
- версионирование;
- безопасную пересборку после изменения мероприятий.

## 2. Источники данных

### SchoolEvent

`SchoolEvent` является основным источником КПВР.

Используются данные:

- название мероприятия;
- описание;
- модуль воспитания;
- уровни образования;
- классы;
- дата начала;
- дата окончания;
- месяц;
- ответственный;
- соисполнители;
- партнер;
- статус;
- количество участников;
- краткий отчет;
- source metadata, если мероприятие импортировано из документа.

### ActivityDirection

`ActivityDirection` используется для дополнительной классификации мероприятий по направлениям деятельности школы.

Примеры направлений:

- патриотическое воспитание;
- профилактика правонарушений;
- ДДТТ;
- работа с родителями;
- волонтерская деятельность;
- ЮИД;
- Движение Первых;
- школьный музей.

ActivityDirection не заменяет модуль воспитания. Направления используются для аналитики, планов деятельности и проверки полноты, но структура КПВР остается сгруппированной по модулям воспитания.

### EventDirectionRelation

`EventDirectionRelation` связывает одно мероприятие с одним или несколькими направлениями деятельности.

Используется для:

- проверки тематического покрытия КПВР;
- формирования дополнительных планов деятельности;
- выявления пробелов по направлениям;
- предотвращения дублирования мероприятий в разных планах.

### Федеральный календарный план

Федеральный календарный план используется как нормативный источник значимых дат и рекомендуемых событий.

Используются данные:

- федеральные памятные даты;
- государственные праздники;
- тематические дни;
- рекомендуемые воспитательные события;
- уровни образования, если они указаны;
- приоритет источника.

Федеральные даты не должны автоматически становиться мероприятиями школы без подтверждения пользователя, если они были извлечены из документа. После подтверждения они могут стать `SchoolEvent` и попасть в КПВР.

### Воспитательная система

Воспитательная система используется для проверки реалистичности и полноты КПВР.

Используются данные:

- школьные традиции;
- воспитательные объединения;
- инфраструктура;
- социальные партнеры;
- активные школьные направления.

### Объединения

Объединения помогают определить, какие мероприятия должны быть отражены в КПВР.

Примеры:

- волонтерский отряд;
- школьный музей;
- ЮИД;
- Юнармия;
- Орлята России;
- Движение Первых;
- школьный спортивный клуб;
- медиацентр;
- театр.

Если объединение активно, но в КПВР нет связанных мероприятий, Builder должен создавать предупреждение о неполноте.

### Партнеры

Партнеры используются для заполнения и проверки мероприятий, связанных с социальным партнерством.

Примеры:

- библиотеки;
- музеи;
- ДК;
- колледжи;
- учреждения спорта;
- учреждения культуры;
- предприятия;
- общественные организации.

Если партнер указан в воспитательной системе, но не используется ни в одном мероприятии КПВР, это должно учитываться как риск неполного отражения социального партнерства.

## 3. Правила формирования КПВР

### КПВР НОО

В КПВР НОО попадают мероприятия, у которых в `SchoolEvent.educationLevels` указан уровень НОО.

Правила:

- включать только мероприятия, относящиеся к НОО;
- если мероприятие относится к нескольким уровням, оно должно отображаться в КПВР НОО и в других соответствующих планах;
- классы должны быть указаны в формате, понятном пользователю: например, `1-4 классы`, `2 класс`, `1А, 1Б`;
- при отсутствии классов показывать предупреждение о неполных данных.

### КПВР ООО

В КПВР ООО попадают мероприятия, у которых указан уровень ООО.

Правила:

- включать мероприятия для 5-9 классов;
- сохранять связь с модулем воспитания;
- отображать сроки и ответственных;
- при наличии направлений деятельности использовать их для проверки покрытия, но не заменять группировку по модулю.

### КПВР СОО

В КПВР СОО попадают мероприятия, у которых указан уровень СОО.

Правила:

- включать мероприятия для 10-11 классов;
- учитывать профориентационные, гражданско-патриотические, социальные и проектные мероприятия;
- проверять наличие ответственных;
- предупреждать, если план содержит слишком мало мероприятий по ключевым направлениям.

## 4. Правила распределения мероприятий

### Распределение по модулям воспитания

Основная группировка КПВР выполняется по модулям воспитания.

Правила:

- каждое мероприятие должно иметь модуль воспитания;
- если модуль не указан, мероприятие помещается в секцию `Модуль не указан` и помечается как требующее проверки;
- пользовательские модули допускаются, если они есть в справочнике модулей;
- неактивные модули не должны использоваться для новых мероприятий без предупреждения;
- порядок модулей должен соответствовать справочнику модулей воспитания.

### Распределение по месяцам

Месяц определяется по дате начала мероприятия.

Правила:

- если указана дата начала, месяц вычисляется автоматически;
- если указан период, месяц берется по дате начала периода;
- если дата отсутствует, мероприятие попадает в секцию `Срок не указан`;
- мероприятия внутри модуля сортируются по дате;
- при одинаковой дате сортировка выполняется по названию.

### Распределение по уровням образования

Одно мероприятие может относиться к одному или нескольким уровням образования.

Правила:

- НОО, ООО и СОО формируются как отдельные проекции одного массива `SchoolEvent`;
- мероприятие не копируется физически между планами;
- если выбраны все уровни, мероприятие отображается во всех трех КПВР;
- если уровень не указан, мероприятие не должно автоматически попадать в КПВР и должно быть помечено как неполное.

## 5. Правила работы с федеральными датами

Федеральные даты используются как нормативный ориентир.

Правила:

- федеральные даты должны иметь источник и приоритет;
- федеральные даты не должны автоматически перезаписывать школьные мероприятия;
- школьное мероприятие может быть связано с федеральной датой;
- если федеральная дата обязательна или рекомендована, но в КПВР нет связанного мероприятия, Builder должен создать предупреждение;
- при конфликте источников применяется приоритет из Federal Documents Authority:
  1. федеральное законодательство;
  2. ФГОС;
  3. ФОП;
  4. федеральная рабочая программа воспитания;
  5. федеральный календарный план;
  6. рекомендации Минпросвещения;
  7. региональные документы;
  8. муниципальные документы;
  9. локальные акты школы.

Федеральный календарный план не должен автоматически создавать operational events без preview, validation, duplicate detection и user confirmation.

## 6. Правила предотвращения дублей

KPVR Builder не должен создавать дубли мероприятий.

Проверка дублей выполняется до отображения и перед импортом document-derived events.

Сравнивать:

- нормализованное название;
- дату или месяц;
- уровень образования;
- классы;
- модуль воспитания;
- sourcePreviewEventId, если мероприятие импортировано из документа;
- importBatchId, если событие было создано в рамках пакетного импорта.

Нормализация названия:

- привести к нижнему регистру;
- убрать кавычки;
- убрать лишние пробелы;
- заменить двойные пробелы одинарными;
- убрать незначимые знаки препинания.

Правила:

- если найдено совпадение по sourcePreviewEventId, повторный импорт запрещен;
- если найдено похожее мероприятие без source metadata, пользователь должен увидеть предупреждение;
- федеральное событие и школьное мероприятие могут быть связаны, но не должны дублироваться;
- одно мероприятие может отображаться в нескольких планах, но хранится один раз.

## 7. Правила проверки полноты КПВР

Проверка полноты должна показывать пользователю, что мешает считать КПВР готовым.

Проверять:

- есть ли мероприятия для НОО;
- есть ли мероприятия для ООО;
- есть ли мероприятия для СОО;
- заполнены ли модули воспитания;
- заполнены ли даты или сроки;
- указаны ли классы;
- указаны ли ответственные;
- покрыты ли обязательные направления воспитания;
- отражены ли активные объединения школы;
- используются ли социальные партнеры;
- учтены ли ключевые федеральные даты;
- нет ли перегрузки отдельных месяцев;
- нет ли пустых месяцев;
- нет ли дублей.

Статусы полноты:

- `ready` — КПВР можно использовать;
- `needs_review` — КПВР сформирован, но есть предупреждения;
- `incomplete` — не хватает обязательных данных;
- `blocked` — невозможно сформировать корректный КПВР.

КПВР не должен считаться готовым, если:

- отсутствуют мероприятия;
- отсутствуют уровни образования;
- большинство мероприятий не имеет дат;
- мероприятия не имеют модулей;
- нет ответственных;
- план состоит только из неподтвержденных preview-записей.

## 8. Правила версионирования

Каждая утвержденная или пересобранная версия КПВР должна быть воспроизводимой.

Версия должна хранить:

- versionId;
- createdAt;
- educationLevel;
- учебный год;
- список eventId, включенных в план;
- snapshot ключевых полей мероприятий;
- источники федеральных дат;
- параметры группировки;
- статус версии;
- пользователя или режим, создавший версию.

Статусы версии:

- `draft`;
- `reviewed`;
- `approved`;
- `archived`.

Правила:

- черновик может пересобираться;
- утвержденная версия не должна изменяться silently;
- при изменении исходных мероприятий нужно показывать, что КПВР требует пересборки;
- пользователь должен иметь возможность сравнить текущую версию с новой сборкой;
- ручные правки не должны теряться без подтверждения.

## 9. Правила пересборки после изменения мероприятий

Пересборка КПВР требуется, если изменились данные, влияющие на план.

Триггеры:

- создано мероприятие;
- изменено мероприятие;
- удалено мероприятие;
- изменен уровень образования;
- изменена дата;
- изменен месяц;
- изменен модуль воспитания;
- изменены классы;
- изменен ответственный;
- изменены связи с направлениями деятельности;
- импортированы мероприятия из документа;
- изменены федеральные даты;
- изменен учебный год;
- изменена активность модуля воспитания.

Правила пересборки:

1. Определить затронутые уровни образования.
2. Определить затронутые месяцы и модули.
3. Показать пользователю, какие разделы КПВР изменятся.
4. Не заменять утвержденную версию без подтверждения.
5. Создать новую версию после подтверждения.
6. Сохранить связь с предыдущей версией.
7. Сохранить source metadata импортированных мероприятий.

## Ограничения MVP

На этапе MVP KPVR Builder должен быть простым, надежным и объяснимым.

Приоритет:

1. корректная сборка НОО, ООО, СОО из `SchoolEvent`;
2. группировка по модулям воспитания;
3. автоматическое определение месяца;
4. предотвращение дублей;
5. проверка неполных данных;
6. экспорт документа;
7. трассировка источников.

Не нужно преждевременно добавлять:

- автоматический импорт федеральных дат без подтверждения;
- сложное объединение дублей;
- отдельные сущности `PlanNOO`, `PlanOOO`, `PlanSOO`;
- копирование мероприятий в отдельные планы;
- скрытую пересборку без preview.

Главное правило: КПВР является проекцией единого реестра мероприятий, а не отдельным независимым документом.
