# Отчёт приёмочной проверки

Дата проверки: 2026-07-23.

## Итог

Технический контур VOSPITANIE_PRO готов к пилотному наполнению реальными данными школы. Для Work Mode создан отдельный бесплатный проект Supabase `vospitanie-pro`, применена серверная схема, включена авторизация и настроено разграничение доступа. Demo Mode остаётся отдельным локальным демонстрационным контуром.

Реальный школьный пилот ещё не проведён: необходимо назначить сотрудников школы, согласовать срок пилота и пройти приёмочный маршрут из `docs/pilot-runbook.md`.

## Supabase

- Проект `vospitanie-pro` создан в регионе `eu-central-1` и имеет статус `ACTIVE_HEALTHY`.
- Применены миграции `initialize_vospitanie_pro_schema` и `harden_membership_roles_and_indexes`.
- Созданы 15 прикладных таблиц; для каждой включён Row Level Security.
- Анонимные права на прикладные таблицы отозваны.
- Security Advisor не обнаружил замечаний.
- Индексы добавлены для всех внешних ключей, отмеченных Performance Advisor. Оставшиеся информационные сообщения об unused indexes ожидаемы для пустой базы.
- Секретный `service_role` не используется в клиенте и не опубликован в Vercel.

## Живая проверка RLS

Проверка выполнена через публичный Supabase API с двумя временными подтверждёнными учётными записями:

1. Пользователи A и B создали отдельные школы и мероприятия.
2. Пользователь A видел только свою школу и не мог изменить данные школы B.
3. После добавления A в школу B с ролью `viewer` чтение стало доступно, изменение осталось запрещено.
4. После повышения роли до `member` изменение мероприятия стало доступно.
5. Пользователь A не смог самостоятельно повысить свою роль или управлять участниками.
6. Анонимный запрос к прикладным данным был отклонён.
7. После проверки временные школы, мероприятия, memberships и учётные записи удалены; контрольные счётчики равны нулю.

## Production

- Основной адрес: `https://vospitanie-pro.vercel.app`.
- В Vercel Production и Preview добавлены чувствительные переменные `NEXT_PUBLIC_DATA_BACKEND`, `NEXT_PUBLIC_SUPABASE_URL` и `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY`.
- После итогового push требуется дождаться статуса `READY` последнего deployment и повторно проверить `/login`, `/demo` и рабочие маршруты.

## Проверенные сквозные сценарии

1. Demo Mode загружает отдельную демонстрационную школу и не изменяет Work Mode.
2. Новый пользователь Work Mode начинает с пустой школы, а не с демонстрационных данных.
3. Документ проходит загрузку, анализ, preview, проверку дублей и только затем явный импорт.
4. История импорта показывает пакет и источник; безопасный откат не удаляет вручную изменённые мероприятия.
5. Мероприятие остаётся единым источником для КПВР, планов, контроля исполнения и отчётов.
6. Рабочая программа собирается из данных школы, хранит версии, показывает изменения и восстанавливает выбранную версию.
7. Проверка соответствия создаёт замечания с переходами к нужным разделам.
8. Рабочая программа экспортируется в DOCX и нативный PDF.

## Изоляция и безопасность

- Demo и локальный Work Mode используют разные ключи localStorage.
- Supabase Work Mode не импортирует `mockAppState` и создаёт уникальную пустую школу для каждого пользователя.
- Импорт не меняет `AppState.events` без подтверждения пользователя.
- Неуверенные и неполные записи остаются в preview.
- Серверная схема использует `owner_id`, `school_memberships`, RLS и отдельную проверку административной роли.
- Work Mode переключается на Supabase только при полном наборе публичных переменных; Demo Mode при этом остаётся локальным.
- Реализованы регистрация, вход, выход и восстановление пользовательской сессии через Supabase Auth.

## Оставшиеся организационные действия

- назначить владельца пилота со стороны школы;
- создать постоянные учётные записи участников;
- согласовать допустимый состав реальных данных;
- пройти приёмочный маршрут школы;
- зафиксировать обратную связь и дату решения о завершении пилота.
