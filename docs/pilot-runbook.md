# Регламент запуска пилота VOSPITANIE_PRO

> Статус на 2026-07-23: отдельный проект Supabase создан, схема и RLS применены, переменные Vercel настроены, техническая проверка изоляции двумя временными пользователями пройдена. Разделы 2–3 ниже остаются регламентом для повторной проверки после изменений схемы. Следующий шаг — назначение постоянных участников и прохождение школьного приёмочного маршрута.

## 1. Границы пилота

Публичный Vercel-контур используется для демонстрации. Реальные школьные данные разрешается вводить только после подключения отдельного Supabase-проекта, авторизации и проверки RLS двумя пользователями из разных школ.

Не использовать существующие несвязанные Supabase-проекты и не хранить реальные персональные данные в Demo Mode.

## 2. Создание серверного контура

1. Создать отдельный проект Supabase с названием `vospitanie-pro`.
2. Применить `supabase/schema.sql` как одну контролируемую миграцию.
3. Проверить Security Advisor и устранить предупреждения о таблицах без RLS или избыточных правах.
4. Получить Project URL и активный Publishable Key.
5. Добавить в Vercel Production:
   - `NEXT_PUBLIC_DATA_BACKEND=supabase`;
   - `NEXT_PUBLIC_SUPABASE_URL`;
   - `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY`.
6. Не добавлять `service_role` в переменные с префиксом `NEXT_PUBLIC_`.

## 3. Обязательная проверка изоляции

1. Создать пользователей A и B.
2. Создать школу A с `owner_id` пользователя A.
3. Создать школу B с `owner_id` пользователя B.
4. Под пользователем A подтвердить CRUD своих мероприятий и документов.
5. Под пользователем A попытаться прочитать и изменить школу B — запрос должен вернуть 0 строк или отказ.
6. Добавить A в `school_memberships` школы B с ролью `viewer`; чтение должно появиться, изменение — остаться запрещённым.
7. Изменить роль на `editor`; разрешить рабочие изменения, но не управление участниками.
8. Сохранить результаты и снимки Security Advisor в журнале пилота.

## 4. Приёмочный маршрут школы

1. Открыть `/launch-readiness` и заполнить паспорт школы.
2. Заполнить воспитательную систему, объединения, партнёров и инфраструктуру.
3. Импортировать один реальный план через `/document-processing`.
4. Проверить preview, дубли, пропущенные записи и подтвердить импорт.
5. Открыть `/import-history`, проверить источник и безопасный откат на тестовом пакете.
6. Проверить карточку импортированного мероприятия.
7. Собрать КПВР и рабочую программу.
8. Запустить `/compliance-check`, исправить одно замечание по прямой ссылке.
9. Экспортировать DOCX и PDF и визуально сверить титульную часть, кириллицу и переносы.
10. Проверить мобильный просмотр основных рабочих страниц.

## 5. Критерии допуска

- тесты, lint, build и smoke проходят;
- production не содержит runtime errors;
- Demo и Work изолированы;
- пользователь одной школы не видит данные другой;
- импорт требует preview и подтверждение;
- откат не удаляет вручную изменённые записи;
- экспортированные документы открываются без восстановления файла;
- назначен ответственный за обратную связь и определена дата завершения пилота.

## 6. Откат

- При ошибке приложения вернуть предыдущий READY deployment в Vercel.
- При ошибке импорта использовать пакетный безопасный откат в `/import-history`.
- Миграции базы не откатывать удалением таблиц; подготовить отдельную обратную миграцию после резервной копии.
- При подозрении на нарушение изоляции остановить ввод данных и отозвать доступ до проверки RLS.
