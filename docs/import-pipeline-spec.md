# Import Pipeline Specification

## 1. Назначение Import Pipeline

Import Pipeline — полный конвейер обработки документов в Воспитание.PRO: от загрузки файла до безопасного импорта подтвержденных данных в рабочие сущности школы.

Главная задача Import Pipeline — превращать документы школы и нормативные материалы в проверяемые preview-данные, не изменяя operational data без явного подтверждения пользователя.

Конвейер должен обеспечивать:

- извлечение текста и структуры документов;
- классификацию документов;
- извлечение сущностей;
- оценку качества данных;
- предварительный просмотр;
- поиск дублей;
- подтверждение пользователем;
- импорт с трассировкой источников;
- возможность будущего отката импорта.

Import Pipeline не должен автоматически создавать мероприятия, объединения, партнеров, инфраструктуру или нормативные требования без preview, validation, duplicate detection и user confirmation.

## 2. Этапы обработки

### 2.1. Загрузка файла

Пользователь загружает документ через интерфейс системы.

Поддерживаемые форматы:

- DOCX;
- PDF;
- XLSX.

На этапе загрузки фиксируются:

- id файла;
- название файла;
- тип файла;
- размер;
- дата загрузки;
- источник загрузки;
- статус обработки.

Файл не должен сразу изменять рабочие данные школы.

### 2.2. Определение типа файла

Система проверяет:

- расширение файла;
- MIME-тип;
- размер файла;
- допустимость формата;
- возможность обработки.

Если формат неизвестен или файл поврежден, обработка должна завершиться контролируемой ошибкой с понятным сообщением пользователю.

### 2.3. OCR при необходимости

Для PDF система должна различать:

- текстовый PDF;
- сканированный PDF.

Если PDF сканированный:

- не пытаться извлекать несуществующий текст;
- установить статус `requires_ocr`;
- показать пользователю, что требуется OCR;
- не выполнять дальнейшую классификацию и извлечение сущностей как из полноценного текста.

На текущем MVP OCR может быть только подготовленным этапом. Подключение OCR не должно ломать существующий pipeline.

### 2.4. Извлечение текста

DocumentTextExtractor извлекает текст из поддерживаемого документа.

Результат:

- полный текст;
- текстовые блоки;
- предупреждения;
- ошибки извлечения.

Извлечение текста не должно содержать бизнес-логики анализа мероприятий, партнеров или нормативных требований.

### 2.5. Классификация документа

DocumentClassifier определяет тип документа rule-based способом.

Возможные documentKind:

- federal_work_program;
- federal_calendar_plan;
- regional_document;
- municipal_document;
- local_school_document;
- school_work_program;
- kpvr;
- extra_activity_plan;
- social_passport;
- development_program;
- normative_document;
- unknown.

Классификация должна сохранять:

- documentKind;
- confidence;
- matchedSignals;
- classifiedAt.

Если уверенность низкая, документ остается `unknown` или получает статус `needs_review`.

### 2.6. Извлечение сущностей

Entity extractors работают только с нормализованными документами.

Возможные извлекаемые сущности:

- мероприятия;
- объединения;
- партнеры;
- инфраструктура;
- традиции;
- нормативные требования;
- ответственные лица.

Извлеченные сущности сначала существуют только как preview records. Они не должны автоматически попадать в `AppState.events` или другие рабочие справочники.

### 2.7. Проверка качества

Каждая найденная запись должна получить оценку качества.

Для мероприятий используются категории:

- REAL_EVENT;
- WORK_FORMAT;
- ACTIVITY_DIRECTION;
- NOISE.

Для других сущностей должны использоваться аналогичные статусы качества:

- high confidence;
- medium confidence;
- low confidence;
- invalid;
- needs_review.

Оценка качества должна объяснять, почему запись считается надежной или сомнительной.

### 2.8. Preview

Preview — обязательный этап перед любым импортом.

Пользователь должен видеть:

- найденную сущность;
- источник;
- фрагмент документа;
- confidence;
- matchedSignals;
- качество;
- предупреждения;
- возможные дубли.

Preview-записи можно фильтровать, проверять и выбирать для импорта.

### 2.9. Duplicate Detection

Перед импортом система должна искать возможные дубли.

Сравнивать:

- нормализованное название;
- дату или период;
- уровень образования;
- классы;
- тип сущности;
- источник;
- sourcePreviewEventId или другой стабильный preview id;
- importBatchId, если запись уже импортировалась ранее.

Если найден дубль, система не должна автоматически создавать новую сущность.

### 2.10. User Confirmation

Импорт выполняется только после явного подтверждения пользователя.

Пользователь должен понимать:

- сколько записей будет импортировано;
- сколько записей будет пропущено;
- какие записи являются дублями;
- какие записи неполные;
- какие записи имеют низкую уверенность.

Низкоуверенные записи должны оставаться в preview и не импортироваться автоматически.

### 2.11. Import

Импорт создает или обновляет рабочие сущности только для подтвержденных записей.

Каждый импорт должен быть пакетным и аудируемым.

ImportBatch должен хранить:

- batchId;
- createdAt;
- documentIds;
- documentNames;
- importedBy;
- previewEntityCount;
- importableCount;
- importedCount;
- skippedCount;
- duplicateCount;
- incompleteCount.

Импортированные сущности должны хранить source metadata.

## 3. Какие сущности могут извлекаться

### Мероприятия

Извлекаются из:

- КПВР;
- рабочих программ воспитания;
- федерального календарного плана;
- планов деятельности;
- отчетов;
- приказов.

Поля:

- название;
- дата;
- месяц;
- уровень образования;
- классы;
- модуль воспитания;
- направление деятельности;
- ответственный;
- партнер;
- source metadata;
- confidence;
- quality category.

### Объединения

Извлекаются из:

- рабочей программы воспитания;
- положений;
- приказов;
- планов объединений;
- отчетов.

Примеры:

- волонтерский отряд;
- школьный музей;
- ЮИД;
- Юнармия;
- Орлята России;
- Движение Первых;
- медиацентр;
- театр;
- школьный спортивный клуб.

### Партнеры

Извлекаются из:

- разделов социального партнерства;
- КПВР;
- отчетов;
- договоров;
- приказов;
- планов совместной деятельности.

Примеры:

- библиотеки;
- музеи;
- ДК;
- колледжи;
- вузы;
- предприятия;
- учреждения культуры;
- учреждения спорта.

### Инфраструктура

Извлекается из:

- паспорта школы;
- рабочей программы воспитания;
- положений;
- отчетов;
- описаний воспитательной среды.

Примеры:

- школьный музей;
- библиотека;
- медиацентр;
- актовый зал;
- спортивный зал;
- центр детских инициатив;
- музейная комната;
- профильные кабинеты.

### Традиции

Извлекаются из:

- КПВР;
- рабочей программы воспитания;
- отчетов;
- повторяющихся школьных мероприятий.

Традиция должна иметь признаки регулярности, общешкольной значимости или устойчивого упоминания в нескольких источниках.

### Нормативные требования

Извлекаются из:

- ФГОС;
- ФОП;
- федеральной рабочей программы воспитания;
- федерального календарного плана;
- региональных и муниципальных документов;
- локальных актов школы.

Нормативные требования используются для проверки соответствия, но не должны автоматически изменять рабочие документы школы.

### Ответственные лица

Извлекаются из:

- мероприятий;
- приказов;
- планов;
- отчетов;
- положений.

Ответственное лицо должно быть связано с контекстом: мероприятием, объединением, направлением или документом.

## 4. Правила качества данных

Каждая извлеченная запись должна иметь:

- confidence;
- matchedSignals;
- qualityScore;
- qualityReason;
- status.

Общие правила:

- конкретные названия важнее общих формулировок;
- записи с датой, ответственным и источником надежнее записей без контекста;
- повторение в нескольких документах повышает доверие;
- заголовки разделов, цели, задачи и названия таблиц не должны импортироваться как сущности;
- слабые или неполные записи должны оставаться в preview;
- пользователь должен видеть причину оценки качества.

Низкое качество означает:

- не импортировать автоматически;
- показать предупреждение;
- запросить ручную проверку;
- сохранить запись только как preview.

## 5. Правила обработки ошибок

Ошибки должны быть управляемыми и понятными пользователю.

Типы ошибок:

- неподдерживаемый формат;
- поврежденный файл;
- слишком большой файл;
- пустой документ;
- сканированный PDF без OCR;
- невозможно извлечь текст;
- невозможно определить структуру;
- низкое качество извлечения;
- ошибка классификации;
- ошибка сохранения;
- конфликт дублей.

Правила:

- ошибка одного документа не должна ломать весь pipeline;
- пользователь должен видеть статус и причину ошибки;
- техническая ошибка должна логироваться;
- документ с ошибкой не должен создавать рабочие сущности;
- повторная обработка должна быть возможна после исправления причины.

## 6. Правила обратимости импорта

Каждый импорт должен быть обратимым на уровне архитектуры.

Для этого требуется:

- ImportBatch;
- source metadata;
- связь imported entity с preview entity;
- дата импорта;
- источник документа;
- стабильный id preview-записи.

Правила:

- событие или другая сущность, созданная импортом, должна знать свой `importBatchId`;
- система должна позволять в будущем удалить все сущности конкретного batch;
- повторный импорт того же preview id не должен создавать дубль;
- ручные изменения пользователя после импорта должны учитываться перед rollback;
- rollback не должен удалять данные, созданные пользователем вручную.

На MVP UI удаления batch может отсутствовать, но структура данных должна быть готова к этой функции.

## 7. Правила хранения source metadata

Каждая импортированная сущность должна хранить происхождение.

Минимальные поля:

```text
sourceDocumentId
sourceDocumentName
sourcePreviewEntityId
sourceType
sourceFragment
sourceConfidence
sourceQualityScore
importBatchId
importedAt
importedBy
```

Для документов:

```text
documentId
fileName
fileType
documentKind
classificationConfidence
matchedSignals
processedAt
validationStatus
qualityScore
warnings
```

Правила:

- пользователь должен понимать, откуда появилась запись;
- source metadata нельзя удалять при обычном редактировании;
- при экспорте или проверке соответствия источник должен быть доступен;
- при повторной обработке документа старая source metadata должна сохраняться для аудита.

## 8. Правила безопасности

Import Pipeline должен быть безопасным для данных школы.

Правила:

- не выполнять код из документов;
- не доверять содержимому файлов;
- проверять расширение и MIME-тип;
- проверять размер файла;
- не импортировать низкоуверенные записи;
- не перезаписывать operational data без подтверждения;
- не смешивать Demo Mode и Work Mode;
- не отправлять документы во внешние AI API без явного отдельного решения;
- хранить ошибки и предупреждения;
- не показывать пользователю ложную готовность данных.

Для Demo Mode:

- demo-документы не должны попадать в work-state;
- demo-импорт не должен менять рабочие данные;
- сброс demo-данных не должен удалять work-state.

Для Work Mode:

- рабочие документы не должны зависеть от DemoSchoolFactory;
- пользователь должен видеть, какие данные являются подтвержденными, а какие только preview;
- импорт должен быть явным действием.

## Ограничения MVP

На этапе MVP Import Pipeline должен обеспечивать:

- стабильную загрузку документов;
- обработку DOCX, PDF и XLSX;
- классификацию документов;
- извлечение preview-сущностей;
- оценку качества;
- отображение preview;
- поиск дублей;
- подтверждение перед импортом;
- source metadata.

Не нужно преждевременно добавлять:

- скрытый автоматический импорт;
- внешний AI-анализ;
- сложный merge дублей;
- автоматическую замену школьных данных;
- импорт низкоуверенных записей;
- обновление КПВР или рабочей программы без подтверждения.

Главное правило: Import Pipeline должен помогать извлекать полезные данные из документов, но не должен создавать operational data без прозрачного и подтвержденного действия пользователя.
