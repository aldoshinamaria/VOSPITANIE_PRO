# Work Program Builder Specification

## 1. Назначение Work Program Builder

Work Program Builder — механизм автоматической сборки черновика рабочей программы воспитания школы на основе данных, уже внесенных или подтвержденных в Воспитание.PRO.

Его задача — не создавать статический шаблон, а собирать документ из связанных сущностей школы:

- паспорта школы;
- воспитательной системы;
- мероприятий;
- КПВР;
- внеурочной деятельности;
- нормативной базы;
- федеральной базы знаний.

Work Program Builder должен формировать объяснимый черновик, показывать источники каждого блока и позволять пользователю редактировать текст без потери возможности повторной сборки.

## 2. Источники данных

### Паспорт школы

Используется для общей характеристики школы.

Данные:

- название школы;
- регион;
- муниципалитет;
- адрес;
- ФИО директора;
- ФИО заместителя директора по ВР;
- учебный год;
- количество обучающихся;
- количество классов;
- базовая инфраструктура;
- социальные партнеры.

### Воспитательная система

Используется для описания уклада, воспитательной среды, объединений, партнерства и инфраструктуры.

Данные:

- воспитательные объединения;
- школьный музей;
- медиацентр;
- театр;
- ЮИД;
- Юнармия;
- Орлята России;
- Движение Первых;
- волонтерский отряд;
- школьный спортивный клуб;
- совет обучающихся;
- инфраструктура;
- социальные партнеры.

### Мероприятия

Используются для описания видов, форм, содержания воспитательной деятельности и школьных традиций.

Данные:

- название мероприятия;
- описание;
- модуль воспитания;
- направления деятельности;
- уровни образования;
- классы;
- даты;
- ответственные;
- партнеры;
- объединения;
- статус;
- источник данных.

### КПВР

Используется как календарное подтверждение воспитательной деятельности.

Данные:

- мероприятия по уровням НОО, ООО, СОО;
- группировка по модулям;
- сроки;
- классы;
- ответственные;
- федеральные даты;
- школьные традиции.

### Внеурочная деятельность

Используется для описания воспитательного потенциала внеурочной деятельности и дополнительного образования.

Данные:

- программы;
- направления;
- уровни образования;
- классы;
- часы;
- педагоги;
- охват обучающихся.

### Нормативная база

Используется для проверки актуальности и соответствия программы.

Данные:

- локальные акты школы;
- приказы;
- положения;
- региональные документы;
- муниципальные документы;
- федеральные документы;
- статусы актуальности.

### Федеральная база знаний

Используется как справочник требований и ориентиров.

Данные:

- федеральные направления воспитания;
- обязательные разделы программы;
- целевые ориентиры;
- требования ФГОС;
- требования ФОП;
- требования федеральной рабочей программы воспитания;
- федеральные календарные события.

## 3. Какие разделы программы собираются автоматически

### Целевой раздел

Может собираться частично автоматически:

- цель воспитания;
- целевые ориентиры по НОО, ООО, СОО;
- связь с федеральными направлениями воспитания.

Источники:

- федеральная база знаний;
- ФГОС;
- ФОП;
- федеральная рабочая программа воспитания.

### Содержательный раздел

Может собираться автоматически в большей части:

- уклад школы;
- воспитательная среда;
- воспитательные объединения;
- социальное партнерство;
- виды, формы и содержание воспитательной деятельности;
- модули воспитания;
- традиции школы;
- связь с КПВР и мероприятиями.

Источники:

- паспорт школы;
- воспитательная система;
- мероприятия;
- КПВР;
- внеурочная деятельность;
- партнеры;
- инфраструктура.

### Организационный раздел

Может собираться частично автоматически:

- кадровое обеспечение;
- нормативно-методическое обеспечение;
- требования к условиям работы;
- система поощрения социальной успешности;
- анализ воспитательного процесса.

Источники:

- паспорт школы;
- нормативные документы;
- воспитательная система;
- контроль исполнения;
- отчеты;
- федеральная база знаний.

### КПВР как приложение

Собирается автоматически из `SchoolEvent`.

Источники:

- мероприятия;
- модули воспитания;
- уровни образования;
- даты;
- ответственные.

### Приложения

Могут собираться частично автоматически:

- КПВР;
- план внеурочной деятельности;
- перечень модулей воспитания;
- сведения о социальных партнерах;
- сведения о воспитательной системе.

## 4. Какие разделы требуют ручного редактирования

Некоторые разделы не должны считаться полностью готовыми без проверки пользователя.

Требуют ручной проверки:

- итоговая формулировка цели воспитания;
- задачи воспитания с учетом особенностей школы;
- особенности контингента обучающихся;
- описание проблемного поля школы;
- анализ воспитательного процесса;
- выводы по итогам года;
- управленческие решения;
- локальные особенности уклада;
- описание рисков;
- формулировки, которые зависят от управленческой позиции школы.

Если данных недостаточно, Builder должен показывать экран готовности, а не выдавать шаблон как готовый документ.

## 5. Правила формирования

## Уклад школы

Уклад школы формируется из реальных данных системы.

Использовать:

- паспорт школы;
- образовательный контекст;
- воспитательную систему;
- инфраструктуру;
- объединения;
- партнеров;
- традиции;
- повторяющиеся мероприятия;
- КПВР.

Правила:

- использовать реальные названия школы, объединений, музея, партнеров;
- не подставлять вымышленные названия;
- не считать пустой шаблон готовым укладом;
- показывать источники каждого абзаца;
- при отсутствии данных показывать, что нужно заполнить.

## Цели

Цели формируются на основе федеральной рабочей программы воспитания и адаптируются к уровню образования.

Использовать:

- федеральные целевые ориентиры;
- ФГОС;
- ФОП;
- уровень образования: НОО, ООО, СОО;
- особенности школы, если они заполнены.

Правила:

- федеральная формулировка имеет приоритет над локальной;
- локальная формулировка может уточнять, но не отменять федеральную;
- цель должна быть проверяемой и связанной с целевыми ориентирами.

## Задачи

Задачи формируются из целей, направлений воспитания и данных школы.

Использовать:

- федеральные направления воспитания;
- школьные приоритеты;
- воспитательную систему;
- мероприятия;
- планы деятельности;
- результаты анализа.

Правила:

- задачи должны быть связаны с реальными направлениями и мероприятиями;
- не создавать длинный список общих формулировок без связи с данными школы;
- если направление не покрыто мероприятиями, задача должна быть помечена как требующая наполнения.

## Направления воспитания

Направления формируются из федеральной базы знаний и школьных данных.

Обязательные направления:

- гражданское;
- патриотическое;
- духовно-нравственное;
- эстетическое;
- физическое;
- трудовое;
- экологическое;
- ценность научного познания.

Правила:

- каждое направление должно иметь источник в федеральной базе знаний;
- желательно иметь связанные мероприятия;
- если направление не покрыто, Builder должен создать предупреждение;
- школьные направления могут дополнять федеральные, но не заменять их.

## Организационный раздел

Организационный раздел формируется из нормативных и управленческих данных.

Использовать:

- паспорт школы;
- кадровые данные, если есть;
- нормативные документы;
- локальные акты;
- объединения;
- инфраструктуру;
- отчеты;
- контроль исполнения.

Правила:

- не формировать как готовый раздел при отсутствии нормативной базы;
- показывать, какие локальные документы отсутствуют;
- связывать кадровое обеспечение с реальными ответственными;
- отмечать неполные данные.

## Анализ воспитательного процесса

Анализ формируется из исполнения мероприятий, отчетов и матрицы воспитательной деятельности.

Использовать:

- статусы мероприятий;
- контроль исполнения;
- отчеты;
- охват обучающихся;
- баланс направлений;
- риски;
- невыполненные мероприятия;
- перегруженные или пустые периоды.

Правила:

- не подменять анализ общим текстом;
- выводы должны опираться на данные;
- если отчетных данных нет, раздел должен быть помечен как требующий ручной работы;
- рекомендации должны быть понятны заместителю директора по ВР.

## 6. Правила трассировки источников

Каждый сгенерированный фрагмент должен иметь карту источников.

Источник может быть:

- паспорт школы;
- конкретное объединение;
- объект инфраструктуры;
- социальный партнер;
- мероприятие;
- КПВР;
- программа внеурочной деятельности;
- нормативный документ;
- федеральное требование;
- федеральный целевой ориентир;
- результат проверки соответствия.

Для каждого абзаца желательно хранить:

```text
paragraphId
text
sourceType
sourceId
sourceTitle
confidence
generatedAt
generatorName
```

Правила:

- пользователь должен видеть, из чего появился текст;
- низкоуверенные источники не должны использоваться как обязательные факты;
- документные preview-записи не должны попадать в рабочую программу без подтверждения;
- если источник удален или изменен, раздел должен быть помечен как требующий пересборки.

## 7. Правила версионирования

Каждая пересборка программы должна создавать новую версию.

Версия должна фиксировать:

- дату пересборки;
- пользователя или режим;
- список измененных разделов;
- источники, повлиявшие на изменения;
- предыдущий текст;
- новый текст;
- статус: draft, reviewed, approved.

Пользователь должен иметь возможность:

- посмотреть историю версий;
- сравнить версии;
- восстановить предыдущую версию;
- понять, какие данные изменили программу.

Правила:

- ручные правки пользователя не должны silently overwritten;
- при пересборке нужно показывать, какие абзацы будут заменены;
- автосгенерированный текст и пользовательский текст должны различаться по статусу;
- утвержденная версия не должна изменяться без явного действия пользователя.

## 8. Правила повторной сборки программы после изменения данных школы

Повторная сборка нужна, если изменились данные, влияющие на программу.

Триггеры:

- изменен паспорт школы;
- добавлено или изменено объединение;
- добавлен или изменен партнер;
- добавлен или изменен объект инфраструктуры;
- добавлено, удалено или импортировано мероприятие;
- изменен КПВР;
- изменена внеурочная деятельность;
- добавлен новый нормативный документ;
- изменилась федеральная база знаний;
- изменены результаты проверки соответствия.

Правила:

1. Builder должен определить затронутые разделы.
2. Пользователь должен видеть, что программа требует пересборки.
3. Пересборка не должна выполняться silently.
4. Перед заменой текста нужно показать preview изменений.
5. После подтверждения создается новая версия.
6. История версий должна сохраняться.

## Ограничения MVP

На этапе MVP Builder должен отдавать предпочтение рабочей функциональности:

- показывать готовность данных;
- собирать черновик;
- показывать источники;
- предупреждать о неполных разделах;
- сохранять возможность ручного редактирования.

Не нужно преждевременно усложнять:

- сложным AI-редактором;
- микрофункциями форматирования;
- автоматическим утверждением текста;
- скрытым импортом данных;
- переписыванием архитектуры.

Главное правило: рабочая программа должна собираться из данных школы, но оставаться проверяемой, объяснимой и управляемой пользователем.
